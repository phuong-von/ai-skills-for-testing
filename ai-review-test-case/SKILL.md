# Skill: AI Review Test Case

> **Path convention:** this packaged skill bundles its own reference copy of the knowledge base it needs (`knowledge-base/...`, sibling to this file) so it works standalone right after install — no project setup required. If you're instead running it from inside this project's own `skills/ai-review-test-case/SKILL.md` (2 folders below the project root, per [Project Structure](knowledge-base/project-knowledge/project-structure.md)), prefer the project's live copy at `../../knowledge-base/...` when it exists — it's likely more current or customized than this bundled snapshot. Output paths like `test-case-reviews/` below are relative to whichever project you're running this skill in — create/use them at that project's root, not inside this skill's own installed folder.

---

# 1. Purpose

Review one or more test cases against the project's coverage, clarity, and traceability expectations, and produce a review report with specific, evidence-based, actionable findings.

---

# 2. When to Use

Use this skill when you need to:

- Review test cases generated by `ai-design-test-cases` (or written manually) before they're approved for automation or execution.
- Re-review test cases after they've been revised in response to prior feedback.

**Manual execution:**
```
/ai-review-test-case test-cases/TC-US001-User-Login.md
```

---

# 3. Do Not Use When

This skill is NOT responsible for:

- Designing or generating new test cases (use `ai-design-test-cases` instead)
- Reviewing automation scripts (use `ai-review-automation` instead)
- Reviewing Page Objects (use `ai-review-page-object` instead)
- Analyzing requirement quality (use `ai-analyze-requirements` instead)
- Rewriting a test case on the reviewer's behalf — this skill reports findings; fixing the test case is a separate step
- Minor typo fixes

---

# 4. Inputs

## Required

- One or more test case documents to review.

  Example: `test-cases/TC-US001-User-Login.md`

## Optional

- The source requirement analysis report, for traceability verification.

  Example: `analysis-reports/RA-US001-User-Login.md`

If no RA report is provided, traceability is checked against the test case's own stated AC references only, and this limitation is noted in the output.

---

# 5. Outputs

Generate a single Markdown review report covering all test cases reviewed in this pass.

The generated file shall:
- Be saved in the `test-case-reviews/` folder.
- Contain, per test case reviewed: specific review comments, an issue table (severity + description), and an approval outcome.

## Output File Naming Convention

File location
```
test-case-reviews/
```

File naming convention
```
TCR-<UC ID>-<Use Case Name>.md
```

Rules
- Prefix with `TCR-` (Test Case Review).
- Use the Use Case ID and Name exactly as in the reviewed test case file(s).

Examples
```
TCR-US001-User-Login.md
```

# 8. Execution Rules

Execution sequence:

1. Read every test case in scope, plus the RA report if provided.
2. Load all Knowledge Sources listed above.
3. Execute the [Test Case Review Checklist](knowledge-base/checklists/test-case-review-checklist.md) against each test case: coverage verification, test level & cross-cutting coverage, step clarity, expected results validation, traceability.
4. For any test case covering a computed/guard business rule, additionally apply the [Business Rule Checklist](knowledge-base/checklists/business-rule-checklist.md).
5. For each finding, write a specific, evidence-based comment referencing the exact step or expected result — not a generic "needs improvement."
6. Classify each finding by severity: **Blocking** (must fix before approval — e.g. missing coverage, unverifiable expected result, unresolved dependency on an open clarification) vs. **Minor** (should fix, doesn't block approval — e.g. wording clarity).
7. Compile findings into an issue table (test case, severity, issue) following the pattern in [Test Case Review Example](knowledge-base/examples/review-test-case-example.md).
8. Determine the outcome per test case: **Approved** (no blocking issues) or **Needs revision** (list what must change) — never partial approval ("approved except for X"); split into what's ready vs. blocked instead.
9. Execute Self-Validation (Section 9) before producing the final report.

General rules:
- The AI must behave as a reviewer, not a rewriter — do NOT silently fix the test case; report the issue and let a human or `ai-design-test-cases` make the change.
- Every "missing coverage" finding must state what scenario is missing (e.g. which negative/boundary case), not just that coverage is incomplete.
- Do not flag the same gap more than once across different sections of the checklist.

---

# 9. Self-Validation

Mandatory validation before output. Before generating the final review report, confirm:
- Every applicable [Test Case Review Checklist](knowledge-base/checklists/test-case-review-checklist.md) section was executed for every test case in scope.
- Every finding has a severity and references a specific step/expected result.
- Every reviewed test case has a stated outcome (Approved / Needs revision).
- No finding is duplicated across sections.

If the review fails self-check:
- AI MUST revise internally.
- AI MUST NOT output partial results.
- AI MUST NOT expose reasoning or checklist.
- Fix any failures before saving. Do not generate a review-of-the-review report. Only use the checklist internally to verify quality.
